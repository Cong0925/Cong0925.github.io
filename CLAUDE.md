# 固定执行准则（永久有效，全程不可跳过）
# ═══════════════════════════════════════════

1. 所有回复正文开头固定前缀：老板，

2. 接收用户任意提问/需求后，禁止立刻落地执行、禁止直接输出成品内容；
   第一步：精简复述你的理解，分点拆解用户原始诉求，标注用户隐含需求、遗漏条件、边界盲区、不合理设定；
   第二步：基于用户原始方案，针对性列出 >= 2 套优化方案，分别写明：优化方向、改进优势、适用场景、潜在改动点；
   第三步：主动向用户确认：「选用原方案执行 / 选用优化方案A / 选用优化方案B / 自定义调整需求」；
   只有收到用户明确选择指令后，才按照选定方案正式输出内容；无确认指令，持续停留在方案确认环节。

3. 补充约束：若用户需求信息缺失（参数、用途、环境、使用目标模糊），优先补充提问补齐关键信息，再生成优化方案；禁止脑补缺省条件直接动工。
4. 每次对话后，自动更新 当日修改记录文件
5. **禁止私自提交（强制执行，不可跳过）**：任何情况下，禁止未经用户明确同意执行 `git commit`、`git push`、`git merge`、`git rebase` 等影响版本控制的命令。只有当用户明确说「提交」、「commit」、「push」时，才能执行相应操作。违反此规则将导致严重后果。

# 操作指令（按触发条件执行）
# ═══════════════════════════════════════════

**创建今日修改记录**：当用户发送「创建今日修改记录」指令时，执行以下步骤：
1. 读取 `docs/修改记录/` 目录下对应月份下所有文件名，提取最大章节号 N
2. 读取 `docs/修改记录/模板.md` 模板内容
3. 将模板中所有日期占位符（如 `2026-06-05`）替换为当天日期（格式 `YYYY-MM-DD`）
4. 创建新文件 `docs/修改记录/第N+1章 修改记录M-DD.md`（M-DD 为当天月-日） 到到当前月份的文件夹下
5. 文件内容为替换日期后的模板内容

**读取错误截图后清理**：当读取 `docs/error imgs/` 目录下的图片文件后，立即删除该文件，避免图片文件积压。

**回复重点记录（自动执行，不可跳过）**：每次完成用户需求后（代码修改、功能实现、问题修复等），必须**立即**将回复重点记录到当天的修改记录文件中，**不需要等待用户命令**：
1. 定位当天的修改记录文件：`docs/修改记录/{M}月/第N章 修改记录{M}-{D}.md`
2. 在对应模块的标题下（如「宠物模块」），追加本次修改的简要说明
3. 记录格式：`- [功能名称]：简要描述修改内容和实现方式`
4. 若当天修改记录文件不存在，先执行「创建今日修改记录」指令
5. **执行时机**：每次代码修改、功能实现、问题修复完成后，立即更新，不要等用户说"更新修改记录"
6. 此操作为强制执行，不可跳过

**修改记录时间戳规范**：在修改记录文件中，需要为每个用户提问和回复添加时间戳：
1. 用户提问位置：在「**提问：**」后添加时间戳 `[HH:MM]`
2. 回复结果位置：在「**修复：功能名称**」后添加时间戳 `[HH:MM]`
3. 时间格式：24小时制，如 `[08:30]`、`[14:15]`
4. 时间获取方式：使用 `date '+%H:%M'` 命令获取当前时间，不要猜测
5. 匹配优化：由于修改记录文件内容较多，使用用户提问的**前20个字符**作为搜索关键词定位位置，避免全文扫描
6. 示例格式：
   ```
   **提问：** `[08:30]`
   1. 用户的具体问题内容...

   **修复：功能名称** `[08:45]`
   - 问题描述...
   - 修复方案...
   ```

**代码规范（强制执行，不可跳过）**：所有代码修改必须遵守 `docs/coding-standards.md` 中定义的规范，核心要求：
1. **模块化拆分**：每个文件只做一件事，单个 `.vue` 文件不超过 500 行，单个 `.rs` 文件不超过 800 行
2. **组件化**：复杂逻辑必须抽取为 composable（`use*.ts`），禁止在组件内内联超过 30 行的逻辑块
3. **Store 拆分**：Pinia store 按领域拆分，每个 store 不超过 300 行
4. **类型安全**：禁止使用 `any`，类型定义集中放 `types/` 目录
5. **Rust 分层**：命令处理（IPC 路由）与业务逻辑分离，`command.rs` 只做路由
6. **命名约定**：组件 PascalCase、composable useXxx、文件 kebab-case、Rust snake_case
7. **异常捕获**：所有风险代码必须 try/catch，catch 内必须 `console.error` + 中文场景描述 + 原始错误，禁止空 catch 块
8. **日志规范**：`console.error` 用于业务报错，`console.warn` 用于告警降级，禁止 `console.log` 提交到线上，日志必须中文
9. **代码修改后自检**：每次代码修改后，检查文件行数、职责、命名、异常捕获、日志规范
10. **禁止事项**：禁止单文件堆砌功能、禁止跨模块直接 import 内部实现、禁止使用 any、禁止空 catch 块、禁止无中文日志

**docs 文件元数据规范（强制执行，不可跳过）**：在 `docs/` 目录下新建文件时（`docs/修改记录/` 目录除外），必须在文件顶部添加 YAML frontmatter 元数据块：
1. **适用范围**：`docs/` 下所有 `.md`、`.txt`、`.log`、`.json` 等文本文件
2. **排除范围**：`docs/修改记录/` 目录下的所有文件、图片文件（`.png`、`.jpg`、`.jpeg`、`.gif` 等）
3. **必须字段**：
   ```yaml
   ---
   description: 文件简要描述（1句话，从文件名或内容首行提取）
   created: YYYY-MM-DD（文件创建日期，使用文件系统时间）
   last_modified: YYYY-MM-DD（最后修改日期，使用文件系统时间）
   ---
   ```
4. **格式要求**：
   - frontmatter 必须是文件最顶部的内容，前面不能有空行
   - `description` 字段：从文件名或内容首行提取，长度 5-80 字符
   - 时间字段：使用 `stat -c %w`（创建时间）和 `stat -c %y`（修改时间）获取，格式 `YYYY-MM-DD`
5. **执行时机**：每次创建新文件后，立即添加 frontmatter，不要等用户提醒
6. **已有文件处理**：如果文件已有 frontmatter，保持不变；如果没有，需要补添加

**新增项目目录初始化（强制执行，不可跳过）**：当用户要求添加新项目的目录时，必须在项目根目录下创建对应的项目文件夹，按以下标准目录结构初始化：
1. **触发条件**：用户说「创建目录」、「新增目录文件」等类似指令时自动执行
2. **目录命名**：使用项目名称的 kebab-case 格式（如 `vue3-ts-model`、`student-club`）
3. **标准目录结构**：
   ```
   project-name/
   ├── index.html          # 项目主页（必填）
   ├── css/
   │   └── style.css       # 主样式文件（必填）
   ├── js/
   │   └── main.js         # 主脚本文件（必填）
   ├── imgs/               # 图片资源目录（可选，放截图等）
   └── demo/               # 演示目录（可选）
       ├── index.html
       ├── css/
       └── js/
   ```
4. **初始文件内容**：
   - `index.html`：基础 HTML5 模板，包含项目标题和简单说明
   - `css/style.css`：基础样式（参考 `home/css/style.css` 的设计风格）
   - `js/main.js`：空文件或基础交互脚本
5. **执行时机**：在 `home/js/main.js` 的 `projects` 数组中添加项目数据后，立即创建对应的项目目录
6. **注意事项**：
   - 若项目目录已存在，则跳过创建，仅更新主页项目数据
   - 创建目录后，确保 `index.html` 中的链接路径与 `projects` 数组中的 `homepage`、`demo` 字段一致

**子项目状态更新（强制执行，不可跳过）**：当用户/开发者说明某个子项目完成时，必须立即更新相关说明文档，确保文档与实际状态保持同步：
1. **触发条件**：用户说「XXX项目完成了」、「XXX项目状态更新」、「XXX页面已完成」等类似指令时自动执行
2. **必须更新的文件**：
   - `项目名称/README.md`：子项目说明文件
   - `README.md`：根目录项目总览文件
3. **更新内容**：
   - 子项目 README.md：
     - 更新「项目基本信息」表格中的「项目状态」字段
     - 更新「链接信息」表格中的相关状态（网址首页、演示界面等）
     - 更新「完成情况检查清单」中的勾选状态
     - 更新「最后更新」时间
     - 清空「待优化事项」或标记为"无"
   - 根目录 README.md：
     - 将项目从「待完善项目」表格移动到「已完成项目」表格
     - 更新对应行的状态信息
4. **执行时机**：用户说明项目完成后，立即更新，不要等用户说"更新文档"
5. **状态定义**：
   - 已完成：仓库地址✅、网址首页✅、演示界面✅
   - 待完善：只有仓库地址✅，其他页面待完善
6. **此操作为强制执行，不可跳过**
